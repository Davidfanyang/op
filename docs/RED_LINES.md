# Live Monitor 上线红线

> ⚠️ **必须严格遵守，无例外**

---

## 红线 1：误报控制测试结果

**没有误报控制测试结果前，不能把 live_monitor 结果直接当质检结论。**

### 具体要求
- 必须通过 `false-positive-control.test.js` 测试套件
- 6类边界用例通过率 > 85%
- 主管确认的误报率 < 20%

### 检查方式
```bash
npm test -- tests/false-positive-control.test.js
```

### 违规后果
- 误报导致员工被错误处罚
- 系统失去信任，后续推广受阻
- 可能引发劳动纠纷

---

## 红线 2：主管复核机制

**没有主管复核前，不能把告警结果直接用于绩效。**

### 具体要求
- 所有告警必须经主管人工确认
- 系统仅提供"参考建议"，不做最终结论
- 复核流程必须留痕（谁、何时、结论）

### 复核流程
1. 系统产生告警 → 发送主管
2. 主管查看对话上下文 → 判断是否属实
3. 主管标记：确认有效 / 确认误报 / 待观察
4. 只有"确认有效"的告警才能进入绩效计算

### 违规后果
- 未经复核的处罚属于程序违规
- 员工可申诉，公司处于不利地位
- 管理层承担连带责任

---

## 红线 3：灰度统计验证

**没有灰度统计前，不能全量推送到正式主管群。**

### 灰度要求
- 只接一个项目
- 只接一个入口
- 只看内部监控
- Telegram 告警先发测试群
- 结果只做观察，不做处罚依据

### 三个必须统计的指标
1. **告警总数** - 了解告警频率是否合理
2. **实际误报数** - 基于主管复核计算误报率
3. **主管认可率** - 确认告警质量是否可接受

### 灰度通过标准
| 指标 | 最低要求 | 理想值 |
|------|---------|--------|
| 主管认可率 | ≥ 70% | ≥ 85% |
| 误报率 | ≤ 20% | ≤ 10% |
| 告警率 | ≤ 30% | ≤ 15% |

### 违规后果
- 全量推送可能导致信息轰炸
- 主管群对告警麻木，真正问题被忽略
- 系统被强制下线，项目失败
