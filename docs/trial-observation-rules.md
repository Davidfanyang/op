# 试运行观察规则（Trial Observation Rules）

> **版本**：v1.0  
> **创建时间**：2026-04-24  
> **状态**：最小可用  
> **目的**：明确每天看哪些指标，如何判断是否继续试运行

---

## 一、每天看哪几个指标

### 1.1 核心观察指标

| 序号 | 指标 | 说明 | 正常值 | 异常值 | 数据来源 |
|------|------|------|--------|--------|---------|
| 1 | **总试运行样本数** | 今天有多少次训练触发了知识注入 | 无上限 | 0（可能开关关闭） | 日志统计 |
| 2 | **知识命中次数** | knowledge_hit_count > 0 的次数 | >= 1 次 | 0 次（应命中但未命中） | 日志命中日志 |
| 3 | **知识注入成功次数** | injected = true 的次数 | >= 1 次 | 0 次 | 日志命中日志 |
| 4 | **回退次数** | fallback = true 的次数 | 0 次 | > 50% 样本（频繁回退） | 日志回退日志 |
| 5 | **错知识次数** | 主管确认命中错知识的次数 | 0 次 | > 20% 样本 | 主管记录 |
| 6 | **主管否定次数** | 主管不认可建议答案的次数 | 0 次 | > 30% 样本 | 主管记录 |
| 7 | **FAQ 新增数** | 今天新增沉淀的 FAQ 数量 | 0~5 条 | 0 条（长期无沉淀需关注） | knowledge_base 表 |

### 1.2 指标计算方式

**从日志统计**：
```bash
# 总样本数（今天所有知识注入日志）
grep "KnowledgeInjectionTrial" runtime/logs/bot.log | grep $(date +%Y-%m-%d) | wc -l

# 知识命中次数
grep "命中日志" runtime/logs/bot.log | grep $(date +%Y-%m-%d) | wc -l

# 知识注入成功次数（injected=true）
grep "命中日志" runtime/logs/bot.log | grep $(date +%Y-%m-%d) | grep '"injected":true' | wc -l

# 回退次数
grep "回退日志" runtime/logs/bot.log | grep $(date +%Y-%m-%d) | wc -l

# 未命中次数
grep "未命中内部试运行" runtime/logs/bot.log | grep $(date +%Y-%m-%d) | wc -l
```

**从数据库统计**：
```bash
# FAQ 新增数（今天）
mysql -u root -e "USE trainer_core; SELECT COUNT(*) as new_faq_count FROM knowledge_base WHERE DATE(created_at) = CURDATE();"

# 知识状态分布
mysql -u root -e "USE trainer_core; SELECT status, COUNT(*) as count FROM knowledge_base GROUP BY status;"
```

**主管手动记录**：
- 错知识次数：主管发现并确认的错知识数量
- 主管否定次数：主管不认可建议答案的次数

## 三、什么情况继续试运行

### 3.1 继续条件（全部满足）

| 指标 | 阈值 | 说明 |
|------|------|------|
| **知识命中率** | >= 70% | 命中的样本 / 总样本 |
| **知识注入成功率** | >= 80% | injected=true / 命中次数 |
| **回退率** | <= 30% | fallback=true / 总样本 |
| **错知识率** | <= 10% | 错知识次数 / 命中次数 |
| **主管认可率** | >= 70% | （命中次数 - 否定次数）/ 命中次数 |
| **平均耗时** | <= 15 秒 | duration_ms 平均值 |

**计算公式**：
```
知识命中率 = 知识命中次数 / 总样本数
知识注入成功率 = 知识注入成功次数 / 知识命中次数
回退率 = 回退次数 / 总样本数
错知识率 = 错知识次数 / 知识命中次数
主管认可率 = （知识命中次数 - 主管否定次数）/ 知识命中次数
```

### 3.2 继续示例

**示例**：
```
总样本数：10
知识命中次数：8
知识注入成功次数：8
回退次数：2
错知识次数：0
主管否定次数：1

计算：
- 知识命中率 = 8/10 = 80% ✅ >= 70%
- 知识注入成功率 = 8/8 = 100% ✅ >= 80%
- 回退率 = 2/10 = 20% ✅ <= 30%
- 错知识率 = 0/8 = 0% ✅ <= 10%
- 主管认可率 = (8-1)/8 = 87.5% ✅ >= 70%

结论：✅ 继续试运行
```

## 四、什么情况必须暂停

### 4.1 立即暂停条件（任一满足）

| 指标 | 阈值 | 处理动作 |
|------|------|---------|
| **错知识率** | > 20% | 立即关闭对应场景白名单 |
| **主管否定率** | > 40% | 暂停试运行，检查知识质量 |
| **回退率** | > 50% | 检查 Qwen3 API 和知识库连接 |
| **平均耗时** | > 30 秒 | 临时关闭知识注入，排查性能 |
| **连续 2 天** | 错知识率升高 | 立即关闭对应场景白名单 |
| **客服反馈** | > 3 名客服强烈不满 | 暂停试运行，收集反馈 |

### 4.2 暂停操作

**一键暂停**：
```bash
# 关闭总开关
export KNOWLEDGE_INJECTION_ENABLED=false

# 重启服务
npm restart

# 验证已关闭
tail -f runtime/logs/bot.log | grep KnowledgeInjectionTrial
# 应看到：未命中内部试运行: trial_disabled
```

**只关闭某个场景**：
```bash
# 从场景白名单中移除
export KNOWLEDGE_INJECTION_SCENARIO_WHITELIST=

# 重启服务
npm restart
```

### 4.3 暂停后处理

**处理流程**：
```
1. 立即暂停（关闭开关）
   ↓
2. 收集问题样本（日志、主管记录）
   ↓
3. 分析根因（错知识、API 超时、知识检索失败等）
   ↓
4. 整改（下线错知识、修复 API、优化检索等）
   ↓
5. 小范围验证（1~2 次训练）
   ↓
6. 观察 30 分钟，无异常则重新启用
```

## 五、什么情况允许扩大范围

### 5.1 扩大场景条件（全部满足）

| 条件 | 说明 |
|------|------|
| **当前场景稳定运行** | >= 7 天无重大问题 |
| **知识命中准确率** | >= 80%（连续 7 天） |
| **建议答案主管认可率** | >= 70%（连续 7 天） |
| **回退率** | <= 20%（连续 7 天） |
| **无明显错知识** | 错知识率 <= 5%（连续 7 天） |
| **主管明确同意** | 主管书面或口头确认 |

### 5.2 扩大人员条件（全部满足）

| 条件 | 说明 |
|------|------|
| **当前人员反馈积极** | 无强烈不满 |
| **无明显问题** | 或问题已解决 |
| **已有完整反馈机制** | 问题记录和跟踪机制 |
| **主管明确同意** | 主管书面或口头确认 |

### 5.3 扩大操作流程

**增加场景**：
```bash
# 1. 修改场景白名单（追加新场景）
export KNOWLEDGE_INJECTION_SCENARIO_WHITELIST=lanton_transfer_success_not_received,new_scenario

# 2. 重启服务
npm restart

# 3. 观察新场景日志
tail -f runtime/logs/bot.log | grep new_scenario

# 4. 观察 3 天，无问题则正式启用
```

**增加人员**：
```
1. 通知新参与人员使用方式
2. 提供问题反馈渠道
3. 持续观察日志 3 天
4. 收集反馈，评估是否继续扩大
```
