# 主管判读规则指南

> 本文档定义 supervisor 在 training 和 live_monitor 场景下的判读规则,帮助主管高效、准确地处理评估结果。

---

## 一、Training 低分时主管怎么判断

### 1.1 判读流程

```
查看分数 → 阅读 Findings → 阅读 Suggestions → 结合 AI 教练总结 → 做出处理决定
```

### 1.2 可见字段说明

| 字段 | 说明 | 判读要点 |
|------|------|---------|
| **score** | 综合评分(0-100) | <30: 严重问题, 30-50: 需要改进, 50-70: 一般, >70: 良好 |
| **findings** | 问题诊断列表 | 关注具体哪个维度出问题(礼貌/专业/流程) |
| **suggestions** | 改进建议列表 | 可直接用于员工辅导 |
| **coach_summary** | AI 教练总结 | 快速了解问题本质 |
| **standard_reply** | 建议回复 | 参考标准话术 |
| **reviewStatus** | 复核状态 | pending → 待处理 |

### 1.3 处理决定及适用场景

| 决定 | 含义 | 适用场景 | 后续动作 |
|------|------|---------|---------|
| **viewed** | 已看 | 问题轻微,已了解情况 | 记录备案,无需跟进 |
| **needs_training** | 需补训 | 话术不规范/流程不熟悉/态度问题 | 安排专项训练,跟踪改进 |
| **passed** | 可放过 | 误判/特殊情况/情有可原 | 标记误报,优化评估规则 |
| **needs_review** | 待复盘 | 复杂案例/需团队讨论/流程缺陷 | 组织复盘会议,优化流程 |

### 1.4 判读示例

#### 示例 1: 话术问题 → needs_training

```json
{
  "score": 45.5,
  "findings": [
    {"dimension": "礼貌程度", "description": "回复过于生硬,缺少礼貌用语"},
    {"dimension": "专业度", "description": "没有使用标准服务话术"}
  ],
  "suggestions": [
    "使用更礼貌的开场白,如'您好,很高兴为您服务'",
    "主动询问客户需求,而不是等待客户提问"
  ],
  "coach_summary": "客服回复过于简短生硬,建议增加礼貌用语和主动服务意识",
  "standard_reply": "您好,很高兴为您服务!请问有什么可以帮助您的吗?"
}
```

**主管判断**: 这是典型的**话术问题**,客服缺乏标准服务用语训练  
**处理决定**: `needs_training` - 安排礼貌用语专项训练  
**备注**: "回复太生硬,需补训礼貌用语"

## 二、Live Monitor 告警时主管怎么判断

### 2.1 判读流程

```
接收告警 → 查看告警级别 → 查看分数和维度 → 阅读原始对话 → 做出复核决定
```

### 2.2 告警级别说明

| 告警级别 | 分数阈值 | 含义 | 响应时效 |
|---------|---------|------|---------|
| **critical** | <20 分 | 严重问题,需立即关注 | 立即处理 |
| **warning** | 20-40 分 | 警告级,需要关注 | 当天处理 |
| **observation** | 40-50 分 | 观察级,建议改进 | 本周内处理 |

### 2.3 复核决定及适用场景

| 决定 | 含义 | 适用场景 | 后续动作 |
|------|------|---------|---------|
| **approved** | 确认告警有效 | AI 判断准确,客服确实有问题 | 记录在案,纳入考核 |
| **rejected** | 拒绝告警 | AI 误判/特殊情况 | 标记误报,优化规则 |
| **needs_edit** | 需要编辑 | 部分正确,需调整评估 | 修改评估结果 |

### 2.4 判读示例

#### 示例 2: 服务态度问题 → approved

```json
{
  "score": 14,
  "alertLevel": "critical",
  "findings": [
    {"dimension": "服务态度", "description": "客服回复'不知道',态度敷衍"}
  ],
  "customerMessage": "我转账成功但对方没收到钱,怎么回事?",
  "userReply": "不知道"
}
```

**主管判断**: 客服面对客户转账问题直接回复"不知道",属于**严重服务态度问题**  
**复核决定**: `approved` - 确认告警有效  
**备注**: "客服回复不规范,需立即辅导"

## 三、哪些问题属于话术问题

### 3.1 话术问题特征

✅ **属于话术问题**的表现:
- 缺少礼貌用语(您好/请/谢谢)
- 回复过于简短生硬
- 没有使用标准服务开场白/结束语
- 语气不够友好但态度没问题
- 缺少主动服务意识

### 3.2 典型场景

| 场景 | 问题回复 | 标准回复 | 问题类型 |
|------|---------|---------|---------|
| 开场白 | "什么问题?" | "您好,很高兴为您服务,请问有什么可以帮助您的?" | 话术 |
| 结束语 | "没了" | "请问还有其他可以帮助您的吗?祝您生活愉快!" | 话术 |
| 询问需求 | "说" | "请您详细描述一下遇到的问题,我会尽快为您处理" | 话术 |

### 3.3 处理建议

- **处理方式**: `needs_training` - 安排话术训练
- **训练重点**: 标准服务用语、礼貌表达、主动服务
- **考核影响**: 轻微,可通过训练快速改进

## 四、哪些问题属于流程问题

### 4.1 流程问题特征

✅ **属于流程问题**的表现:
- 客服不知道如何处理特定业务
- 缺少关键业务信息(如查询步骤、处理时效)
- 没有按照标准流程操作(如未核实身份、未记录工单)
- 给出的解决方案不符合公司规定

### 4.2 典型场景

| 场景 | 问题回复 | 标准流程 | 问题类型 |
|------|---------|---------|---------|
| 转账查询 | "不知道" | 1.核实身份 2.查询交易记录 3.告知客户处理进度 | 流程 |
| 退款申请 | "你等几天" | 1.记录退款原因 2.提交申请 3.告知处理时效(3-5工作日) | 流程 |
| 投诉处理 | "这不是我的问题" | 1.安抚情绪 2.记录投诉内容 3.转交相关部门 | 流程 |

### 4.3 处理建议

- **处理方式**: `needs_review` - 组织流程复盘
- **改进重点**: 业务流程培训、SOP 优化、知识库完善
- **考核影响**: 中等,需要系统性改进

## 五、哪些问题不适合直接作为员工考核依据

### 5.1 不适合考核的情况

❌ **不适合直接考核**的情况:

#### 1. AI 误判
- **表现**: 客服回复实际没问题,但 AI 评分过低
- **原因**: 模型理解偏差、场景匹配错误、阈值设置过严
- **处理**: 标记 `passed`,记录误报原因,优化评估规则

#### 2. 特殊情况
- **表现**: 客服回复看似不规范,但有合理原因
  - 客户情绪激动,客服需要快速响应
  - 系统故障,客服在紧急处理
  - 重复问题,客户多次询问同一问题
- **处理**: 标记 `viewed` 或 `passed`,备注特殊情况

#### 3. 知识库缺失
- **表现**: 客服不知道答案,因为知识库没有覆盖
- **原因**: FAQ 不完整、新业务未更新、场景未录入
- **处理**: 标记 `needs_review`,补充知识库,不考核员工

#### 4. 流程缺陷
- **表现**: 客服按照现有流程操作,但流程本身有问题
- **原因**: SOP 过时、流程不清晰、权限不足
- **处理**: 标记 `needs_review`,优化流程,不考核员工

#### 5. 系统问题
- **表现**: 客服因为系统限制无法提供更好服务
- **原因**: 系统响应慢、功能缺失、权限限制
- **处理**: 标记 `passed`,反馈技术团队,不考核员工

### 5.2 判断原则

**主管判断是否适合考核的核心原则**:

```
问题是否在员工可控范围内?
  ├─ 是 → 可以考核 (话术、态度、流程执行)
  └─ 否 → 不考核 (系统、流程、知识库、特殊情况)
```

### 5.3 误报处理流程

```
发现误报 → 标记 passed → 记录误报原因 → 定期分析误报率 → 优化评估规则
```

**常见误报原因**:
- `threshold_too_sensitive` - 阈值过严
- `model_understanding_limitation` - 模型理解局限
- `scenario_match_error` - 场景匹配错误
- `other` - 其他原因

## 六、主管判读决策树

```
收到评估结果
  │
  ├─ Training 模式
  │   │
  │   ├─ 分数 < 30?
  │   │   ├─ 是 → 话术问题? → needs_training
  │   │   │                    (安排专项训练)
  │   │   └─ 否 → 流程问题? → needs_review
  │   │                        (组织复盘)
  │   │
  │   ├─ 分数 30-50?
  │   │   ├─ 首次出现? → needs_training
  │   │   └─ 反复出现? → needs_review
  │   │
  │   └─ 误判/特殊情况? → passed
  │
  └─ Live Monitor 模式
      │
      ├─ critical 告警?
      │   ├─ 态度问题? → approved
      │   │               (纳入考核)
      │   ├─ 流程问题? → needs_review
      │   │               (复盘流程)
      │   └─ AI 误判? → rejected
      │                  (标记误报)
      │
      ├─ warning 告警?
      │   ├─ 确认问题? → approved
      │   └─ 情有可原? → rejected
      │
      └─ observation 告警?
          ├─ 建议改进 → approved
          └─ 可接受 → rejected
```

## 七、主管操作清单

### 7.1 日常操作

- [ ] 查看 training pending 列表: `node scripts/training-queue-processor.js list`
- [ ] 查看单条详情: `node scripts/training-queue-processor.js detail <review_id>`
- [ ] 执行处理: `node scripts/training-queue-processor.js process <review_id> <decision> <reviewer> [comment]`
- [ ] 查看统计: `node scripts/training-queue-processor.js stats`

### 7.2 判读要点

- [ ] 先看分数,再看 findings
- [ ] 区分话术问题和流程问题
- [ ] 判断是否在员工可控范围内
- [ ] 特殊情况备注清楚
- [ ] 误报及时标记,优化规则

### 7.3 定期复盘

- [ ] 每周查看 training stats,识别高频问题
- [ ] 分析误报率,优化评估阈值
- [ ] 总结典型案例,更新培训材料
- [ ] 反馈流程/系统问题给相关部门

## 八、常见问题 FAQ

### Q1: 分数多少算低分?
- <30 分: 严重问题,必须处理
- 30-50 分: 需要改进,建议处理
- 50-70 分: 一般水平,可选处理
- >70 分: 良好,无需处理

### Q2: 如何区分话术问题和态度问题?
- **话术问题**: 客服想做好,但不知道怎么表达 → needs_training
- **态度问题**: 客服敷衍、不耐烦、不负责 → approved (纳入考核)

### Q3: 什么时候用 passed,什么时候用 rejected?
- **training 模式**: 用 `passed` (可放过)
- **live_monitor 模式**: 用 `rejected` (拒绝告警)

### Q4: 误报率高怎么办?
- 检查阈值设置是否过严
- 检查场景匹配是否准确
- 收集误报案例,优化评估规则
- 定期调整模型参数

### Q5: 如何判断是否是流程问题?
- 客服是否按照现有 SOP 操作?
- SOP 是否覆盖了这个场景?
- 客服是否有权限/资源解决问题?
- 如果 SOP 本身有问题 → 流程问题,不考核员工
