# AI 用户模拟器实现报告

## 任务概述

为 trainer-core 的 TG 训练系统实现"AI 用户模拟器"，让训练系统可以模拟真实用户多轮对话，而不是单轮 FAQ 问答。

## 完成日期

2026-04-17

## 实现内容

### 1. 核心模块：services/user-simulator.js

**文件路径：** `/Users/adime/.openclaw/workspace/trainer-core/services/user-simulator.js`

**职责：**
- ✅ 根据场景生成第一轮用户问题
- ✅ 根据 conversation 继续生成后续用户发言
- ✅ 控制轮数在 3~6 轮之间
- ✅ 支持结束判断

**输入协议：**
```javascript
{
  project: string,           // 项目标识
  scenario: object,          // 场景对象（含 stages）
  conversation: array,       // 当前对话历史 [{role: "user"|"agent", content: string}]
  round: number              // 当前轮次（从0开始）
}
```

**输出协议：**
```javascript
{
  user_message: string,      // 生成的用户消息
  is_finished: boolean       // 是否应该结束对话
}
```

**核心特性：**

1. **第一轮消息生成**
   - 优先使用场景的 `customerMessage`
   - 如果没有，则根据场景标题和描述生成自然开场白
   - 确保表达自然、口语化

2. **后续消息生成**
   - 优先使用本地模型生成（当 `USE_LOCAL_MODEL=true`）
   - 自动降级到规则版（本地模型不可用时）
   - 基于完整对话上下文
   - 考虑当前所处阶段
   - 根据客服回复质量调整用户反应

3. **结束判断**
   - 最少3轮才允许结束
   - 最多6轮强制结束
   - 客服表现优秀（pass）且无问题/缺失，至少3轮后可提前结束
   - 场景的所有阶段已完成，至少3轮后可结束

4. **用户表达控制**
   - ✅ 口语化、简洁
   - ✅ 可能有错别字或不完整句子
   - ✅ 根据客服回复做出反应
   - ✅ 逐步透露信息，不一口气说完
   - ❌ 不使用敬语或过于正式的表达
   - ❌ 不像客服或AI的语气
   - ❌ 长度控制在 10-50 字

**实现方式：**

- **模式A：本地模型生成**（推荐）
  - 调用 `http://localhost:8001/score`
  - 基于详细提示词生成
  - 表达更自然、多样化

- **模式B：规则版生成**（降级方案）
  - 基于场景阶段生成
  - 根据分析结果调整（追问/抱怨/确认）
  - 保证基本的对话逻辑

## 技术实现细节

### 架构设计

```
┌─────────────────────────────────────┐
│   Telegram Bot (未来接入)            │
│   - 接收客服回复                     │
│   - 发送用户消息                     │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│   Session Store                     │
│   - 保存 conversation               │
│   - 保存 round                      │
│   - 保存 analysisResult             │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│   User Simulator (本次实现)          │
│   - 生成用户消息                     │
│   - 控制对话轮数                     │
│   - 判断对话结束                     │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│   Evaluation Service                │
│   - 分析客服回复                     │
│   - 返回 analysisResult             │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│   Core Analysis Pipeline            │
│   - 场景识别                         │
│   - 阶段判断                         │
│   - 质量评估                         │
└─────────────────────────────────────┘
```

### 数据流

```
1. 训练开始
   ↓
2. UserSimulator.generateUserMessage(round=0)
   → 输出: user_message (第一轮问题)
   ↓
3. 客服回复 (Telegram 输入)
   ↓
4. EvaluationService.evaluate()
   → 输出: analysisResult
   ↓
5. UserSimulator.generateUserMessage(round=1, analysisResult)
   → 输出: user_message (第二轮问题)
   ↓
6. 重复步骤 3-5，直到 is_finished=true
   ↓
7. 训练结束，输出总结
```

### 降级策略

```
本地模型可用？
  ├─ 是 → 使用本地模型生成（更自然）
  └─ 否 → 降级到规则版（保证功能）
           └─ 基于场景阶段和分析结果生成
```
